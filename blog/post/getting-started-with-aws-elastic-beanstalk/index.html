Getting started with AWS Elastic Beanstalk

AWS Elastic Beanstalk is a fully managed service designed to streamline the development lifecycle, increase uptime and improve productivity.

Elastic Beanstalk is a platform as a service (PaaS) provided by Amazon Web Services, built to simplify the process of provisioning backend architecture required to run modern, complex web applications. It is a fully managed service designed to streamline the development lifecycle, increase uptime, and overall, improve productivity. For startups, small teams, and do-it-yourselfers it is the perfect solution as it doesn't require a ton of maintenance or a degree in computer science to get it working.

What is Elastic Beanstalk?

Elastic Beanstalk simplifies the process of provisioning servers, load balancers, and other necessary cloud services for your backend web application. Instead of configuring your backend, you - the developer - should be spending your time doing what you do best. Writing code and pushing out new features! That is what Elastic Beanstalk aims to help you do. With multitudes of options and integrations, you are able to quickly get your code from your workstation to the cloud and into production, reducing development time and increasing productivity.

What are the main benefits to using Elastic Beanstalk?

The most obvious benefit to using aws eb is its ease of use and its ability to drastically reduce development time. As a single developer or a small company with no dedicated dev-ops team, you need to be able to get your code onto a server somewhere so that it can be consumed by a client.

In the old days this would mean you would need to manually configure a server (maybe even a physical one). If you wanted to implement load balancing you would need to configure multiple servers and have knowledge of advanced networking in order to configure NGINX or Apache Tomcat. And let's not forget ssh keypairs. I hope you didn't skip your lecture on Linux terminal commands and security protocols!

Elastic Beanstalk abstracts all this away from the developer so you only have to worry about writing your code and let Jeff Bezos handle all the complex architectural boilerplate!

What are the drawbacks of Elastic Beanstalk?

Well with every bit of code running anywhere in the world there are always going to be bugs. And that is no exception with AWS Beanstalk. For it's simplicity there is a price to be paid and that is reliability. Sometimes deployments work, sometimes they don't! If you've had some experience with AWS web hosting or any AWS services for that matter, you will know just how long deployments can take. Especially when you have configured multi-tenancy or load balancing.

I've personally spent days rebuilding broken environments and waiting for rollbacks to complete. It's really not fun. Much of the time there is no indication as to why deployments fail so you are left guessing as to what the problem is. There are logs available but even getting the logs can prove challenging - and sometimes impossible to do on a broken environment. If you've ever had the dreaded: "Environment named xxxxxx is in an invalid state for this operation. Must be ready." error - you will know what I mean!

Should I use it anyway?

Well I think you know the answer to this one but yes! Yes you should use Elastic Beanstalk. It is an incredible tool, so easy to use and regardless of its flaws has saved me so much time and headache. As a full stack web developer and entrepreneur, it provides exactly what I need. Ease of use and reduced development time. I don't need a team of architects to keep it running and I'm able to quickly implement security, network rules, load balancing, ssl, and have the ability to quickly scale up or scale down as required.

How to configure a node.js web server environment on AWS Elastic Beanstalk

And now the fun part! Here I will show you how to use Elastic Beanstalk. In this tutorial I will be using Node.js but the steps are the same to deploy using any platform.

Step 1 - Create a new environment

Navigate to Elastic Beanstalk in the AWS console and create a new environment.

Create an elastic beanstalk environment

Step 2 - Select environment tier

If your environment needs to have a public facing url then you should select web server environment. If your application will only sit isolated in your backend and consume from a queue then you should select worker environment. In this case I am going to select web server environment.

Select elastic beanstalk environment tier

Step 3 - Name the application and select platform

Give the application and environment a unique name to identify them by and then select your desired platform. I am using Node.js here but you should of course select whatever language your code is written in.

Select a managed platform

Step 4 - Configure environment variables and log streaming

Click the 'Edit' button in the software section of the page to input your applications environment variables. Keep in mind that if you deploy your app with invalid env vars, your code will likely crash and cause eb to automatically rollback to a previous application version. If you would like to enable log streaming you can do it here.

Configure environment variables

Step 5 - Configure auto scaling and load balancing

Herein lies the magic of Elastic Beanstalk. Click to edit the capacity options. You will be presented with a form to configure your auto scaling group which consists of a load balancer and a number of ec2 instances of varying sizes. Here you will select an environment type of Load balanced and select a couple instance sizes which you would like to use.

Consider this chart to decide what suits your needs. I will recommend using both t2 (or t3) micro and small to start with. EB will initialize your environment with the smallest instance type you have selected and when a certain limit is reached in terms of processing power or network activity will automatically scale up to the next desired instance size or add another instance to the pool.

Leave the AMI as default unless you know what you are doing! If you would like to know more I have a guide on building and using a custom AMI.

Configure auto scaling group

Step 6 - Add an ssh keypair

This step is required if you are planning on using SSH to interface with the underlying Elastic Compute instance. If you already have a keypair you can select it here. If not you will need to go to the ec2 console and create a new one.

Modify security and add key pair

Step 7 - Configure application load balancer (alb)

Here make sure you have selected application load balancer. To enable SSL you must add a new listener.

Configure application load balancer

Step 8 - Modify listener configuration

Select protocol HTTPS and set the port to 443. Select your SSL certificate here and SSL policy. AWS documentation recommends using ELBSecurityPolicy-2016-08 for compatibility so that is what I will do here. Click to add the listener and then click save.

Modify application load balancer listener

Step 9 - Configure network settings

Navigate to network options and click the button to edit. Here you must set your visibility to public if you wish for your environment to be accessible over the internet. The more availability zones and subnets you chose here the more you stand to benefit from automatic failover.

Configure eb environment network settings

Remember what I said about how long these deployments can take? Go get some popcorn it's gonna be a while.

Once you see a big green checkmark you know you have been successful!
