Containerized Application Architecture With Docker

There are many benefits to building your applications into containers. Portability being the first thing that comes to mind.

There are many benefits of building your applications into a Docker container. The most obvious being portability. But what does it really mean for an application to be portable?

Container ship setting off from the port

For an application to be portable - it must be capable of being run on any machine. This is achieved by bundling the software together with a virtual operating system. The operating system used will usually be a flavor of Linux which will be capable of performing a specific task.

Keep in mind that while similar, a Docker container is in fact not a virtual machine. While a container is capable of running an entire operating system, the key differences between VM's and and Docker containers is the manner in which they allocate system resources.

A virtual machine will have access to an allotted portion of the host machines physical hardware and run completely isolated from the host machine. If a user were to spin up multiple VM's on the same device - many resources would be consumed and the device may not function properly and will be operating at limited capacity.

In contrast - a single instance of Docker will act as a host for any number of containers - allowing many containers to be run at once. The containers do not require allocation of hardware resources because the Docker service running on the host machine has already allocated all the resources it will need to manage the running of each container. This allows a developer to run many containers on the same host OS without a problem.
